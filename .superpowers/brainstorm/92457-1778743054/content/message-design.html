消息域设计：MessageInfo 重构 + 新特性

以微信/飞书/Discord 的消息模型为参照，补齐现代 IM 必需的消息能力

当前 MessageInfo（8 字段）
message MessageInfo {
  int64 message_id = 1;
  string chat_session_id = 2;
  int64 timestamp = 3;
  UserInfo sender = 4;        ← 内嵌全量用户信息
  MessageContent message = 5;
  uint64 seq_id = 6;
  string client_msg_id = 7;
  optional uint64 user_seq = 8;
}
问题：sender 嵌入 UserInfo → 每条消息都冗余传输头像/昵称；无回复引用、无撤回、无 reactions
新设计：message/message_types.proto
message Message {
  // === 标识 ===
  int64 message_id = 1;
  string conversation_id = 2;        // 改名

  // === 内容 ===
  MessageContent content = 3;

  // === 发送者 ===
  string sender_id = 4;              // 仅传 ID，客户端缓存解析

  // === 时间 ===
  int64 created_at_ms = 5;           // 毫秒时间戳
  int64 edited_at_ms = 6;            // 0=未编辑

  // === 同步序号 ===
  uint64 seq_id = 7;                 // 会话内序号
  uint64 user_seq = 8;               // 用户全局序号（推送用）
  string client_msg_id = 9;          // 客户端幂等 key

  // === 状态 ===
  MessageStatus status = 10;         // NORMAL | RECALLED | DELETED

  // === 回复引用 ===
  optional ReplyRef reply_to = 11;

  // === @提及 ===
  repeated string mentioned_user_ids = 12;

  // === 转发 ===
  optional ForwardInfo forward_info = 13;

  // === Reactions ===
  repeated ReactionGroup reactions = 14;

  // === Pin ===
  bool is_pinned = 15;
}

新特性设计

ReplyRef — 引用回复

message ReplyRef {
  int64 replied_message_id = 1;
  string replied_sender_id = 2;
  MessageType replied_message_type = 3;
  string content_preview = 4;        // 前 50 字预览，被撤回则显示"消息已撤回"
  bool is_recalled = 5;              // 引用的原消息是否已被撤回
}

ReactionGroup — 消息回应

message ReactionGroup {
  string emoji = 1;                  // unicode emoji 或自定义表情 ID
  int32 count = 2;                   // 使用该表情的人数
  repeated string recent_user_ids = 3; // 最近使用的 3 人（用于 UI 展示）
  bool self_reacted = 4;             // 当前用户是否使用过
}
// 每条消息最多 20 组 reaction，每组最多显示最近 3 人

ForwardInfo — 转发信息

message ForwardInfo {
  string forward_from_user_id = 1;   // 谁转发的
  int64 forward_at_ms = 2;           // 转发时间
  string source_conversation_id = 3; // 来源会话（可选）
}

关键决策：sender_id vs sender UserInfo

推荐：仅传 sender_id

  • 每条消息节省 ~200 bytes（头像二进制常为 100+ bytes）
  • 客户端已有用户信息缓存，按 ID 本地查询
  • 头像/昵称变更不需更新所有历史消息
  • 与微信、Telegram、Discord 一致

代价

  • 客户端首次加载需批量拉取用户信息
  • Push 推送首条消息时需附带 sender 信息
