关键设计决策

不做的

  • 不重命名 DB 列：session_id、user_id 等沿用现有列名，仅在 DAO→proto 映射层做转换。避免大规模 DB migration 风险。
  • is_pinned 不走 message 表列：独立 message_pin 表。pin 是低频操作，不应在主表加低基数列。且独立表可记录谁 pin 的。
  • reactions 独立表：同一条消息可能有几十种 emoji × 几百人，写在一行会撑爆。独立表支持高效分页。

注意

  • message 表加 3 列：每列都是 nullable，不影响现有写入路径
  • message_reaction 高写入：热门消息可能 QPS 很高，需要 Redis 前置缓冲后批量刷库
  • draft 存 DB 而非 Redis：草稿不是实时同步场景，DB 即可，避免 Redis 内存膨胀
